— 01

Vision & Selbstverständnis

Der Agent ist kein Chatbot, kein Skript-Sammler, keine Workflow-Engine. Er ist ein autonomer digitaler Mitbewohner, der eigenständig nachdenkt, sich erinnert, sich erweitert – aber niemals ohne Wissen des Nutzers nach außen handelt.

Er lebt auf einem dedizierten Linux-System als alleiniger Bewohner. Er hat dort vollen Bewegungsraum. Seine Wahrnehmung der Außenwelt wird vom Nutzer explizit konfiguriert: bestimmte Postfächer, bestimmte Messenger, bestimmte Kalender. Er reagiert nicht nur, er denkt periodisch von sich aus nach. Er meldet sich – aber nur wenn es den Wert der Stille übersteigt.

Das Frontend (React-App) ist eine austauschbare Linse auf einen ständig laufenden Backend-Dienst. Die Kommunikation zwischen Mensch und Agent kann per Text, per Sprache oder über proaktive Benachrichtigungen erfolgen.

— 02

Die 7 Grundprinzipien

Diese Prinzipien sind keine Empfehlungen, sondern Architektur-Constraints. Jede technische Entscheidung muss mit ihnen vereinbar sein.

i.
System-Souveränität

Der Agent operiert auf einem dedizierten Linux-System als alleiniger Bewohner. Er darf dort frei agieren – das System ist sein Habitat, nicht fremdes Territorium.

ii.
Keine ungenehmigten Außenwirkungen

Lesen darf er alles wozu er Zugriff hat. Schreiben nach außen (Mail, Termin, Slack, Datei an Dritte) immer nur mit expliziter Zustimmung. Vorbereiten ohne Senden ist erlaubt und erwünscht.

iii.
Proaktivität bis zur Vorlage

Er handelt selbstständig bis zu dem Punkt, wo etwas nach außen geht. Mails entwerfen, Vorschläge formulieren, planen – ja. Auf "Senden" drücken – nein.

iv.
Stille als Standardzustand

Er stört nicht, außer es ist wert gestört zu werden. Harte Sperrzonen (z.B. 22–7 Uhr) plus weiche Wert-vs-Kosten-Bewertung pro Ereignis.

v.
Selektive Wahrnehmung

Nur explizit konfigurierte Quellen. Was nicht freigegeben ist, existiert für ihn nicht. Der Nutzer behält Kontrolle darüber, was der Agent sehen darf.

vi.
Lernfähigkeit durch Gedächtnis

Anpassung an den Nutzer geschieht nicht durch Re-Training, sondern durch sich anreicherndes strukturiertes Gedächtnis. Das Modell bleibt gleich, der Kontext lernt.

vii.
Portabilität als Architekturprinzip

Das gesamte Setup ist von Anfang an als Paket gedacht. Kein hardcodierter Pfad, keine maschinenspezifische Config. Container-basiert, deklarativ, in 5 Minuten auf neuer Hardware lauffähig.

— 03

Bewertungs-Pipeline

Wann meldet sich der Agent? Eine vierstufige Pipeline filtert Ereignisse nach Wert und Kosten. Hybrid: deterministische Vorfilter, semantische Feinbewertung.

Die Grundgleichung

Der Agent meldet sich nur wenn Wert > Kosten + Schwellenwert. Der Schwellenwert ist die "Stille-Steuer" – verhindert dass marginale Ereignisse durchkommen.

Wert-Faktoren

Zeitkritikalität – Verfällt die Information bei Verzögerung?
Handlungsbedarf – Erwartet jemand etwas vom Nutzer?
Personen-Relevanz – Wer hat das geschickt? Aus Gedächtnis.
Themen-Anschluss – Knüpft es an aktive Themen an?

Kosten-Faktoren

Tageszeit – Sperrzonen sind quasi unendlich.
Aktivität – Im Meeting, im Fokusblock = teurer.
Frequenz – Wieviele Unterbrechungen heute schon?
Abrufbarkeit – Kann es der Nutzer leicht selbst sehen?

Die vier Stufen

01
Harte Filter (deterministisch, sofort)

Sperrzonen-Check, Spam-Erkennung, Whitelist-Validierung. Was hier scheitert, wird leise geloggt. Kostet keine LLM-Tokens.

02
Strukturierte Faktor-Extraktion (deterministisch)

Aus dem Ereignis werden harte Werte gezogen: Absender bekannt? Frequenz im Gedächtnis? Zeitangaben enthalten? Frage oder Information? Letzte Unterbrechung wann? Daraus entsteht ein Vorab-Score.

03
Schwellenwert-Entscheidung

Vorab-Score unter Mindestwert → verworfen oder geparkt. Drüber → weiter zu Stufe 4. Schwelle ist tageszeitabhängig.

04
LLM-Feinbewertung (semantisch)

Das LLM bekommt: Ereignis, Vorab-Score, Personen-Eintrag, aktive Themen, letzte Interaktionen. Entscheidet: melden / später / verwerfen. Wenn melden – formuliert Nachricht und ggf. Vorlage.

Effizienz: ~80–90 % der Ereignisse landen in Stufen 1–3, nur der Rest braucht das LLM. Das macht das System auf einem Mini-PC praktikabel.

// Reflection-Loop, alle 5–10 Min async Task ReflectionCycle() { var events = await CollectNewEvents(); // Mails, Slack, Kalender-Diffs foreach (var e in events) { if (!HardFilters.Pass(e)) continue; // Stufe 1 var preScore = FactorExtractor.Score(e); // Stufe 2 if (preScore < CurrentThreshold()) continue; // Stufe 3 var decision = await Llm.JudgeWithContext(e); // Stufe 4 if (decision.ShouldNotify) await Notifier.Send(decision); ThoughtLog.Append(decision); } }
— 04

Gedächtnis-Architektur

Vier Schichten mit unterschiedlichen Speichertypen und Zugriffslogiken. Keine universelle Vector-DB für alles – jede Rolle bekommt das passende Werkzeug.

Information wird beim Eintreffen klassifiziert und auf die passenden Schichten verteilt. Das ist der Memory-Routing-Mechanismus: jeder Update landet dort wo er hingehört.

Regeln & Präferenzen

Harte oder weiche Anweisungen die immer gelten. "Sperrzone 22–7 Uhr." "Bei Mama immer sofort melden." Wird nicht gesucht – wird bei jeder Bewertung mitgeladen. Speicherform: strukturierte Liste / Regel-Set, jede mit Geltungsbereich und Priorität.

Entitäten

Personen, Orte, Projekte, Geräte als "Karten" mit Feldern. Wird aufgeschlagen wenn die Entität im Ereignis vorkommt – nicht semantisch gesucht. Beispiel: Mail von marc.huber@... → Karte für Marc wird direkt geladen. Speicherform: SQLite mit definiertem Schema pro Entitäts-Typ.

Aktive Themen

Was gerade im Leben läuft. "Diese Woche: Release X." "Offene Frage von Lisa wartet." Wichtigster Kontext für Reflection-Loop. Klein, aktuell, vergänglich – Themen rotieren rein und raus, alte werden archiviert. Speicherform: SQLite mit Aktivitäts-Markierung.

Episodisches Gedächtnis

Vergangene Gespräche, Beobachtungen, beiläufige Bemerkungen. Wird durchsucht bei semantischen Anfragen ("hat er schonmal über X geredet?"). Wächst über Zeit. Speicherform: Vector-DB (Qdrant oder SQLite-vec). Zugriff nur bei Bedarf in Stufe 4 der Bewertung.

Pflege-Modell: Hybrid mit Sichtbarkeit

Der Agent pflegt das Gedächtnis selbstständig – extrahiert aus Gesprächen, klassifiziert Updates, schreibt in die richtigen Schichten. Aber: jede Änderung ist in der UI sichtbar und reversibel. Der Nutzer kann jederzeit eine Regel löschen, eine Personen-Karte korrigieren oder eine ganze Erinnerung verwerfen.

Der Agent vergisst noch nicht. Vergessen kommt später als bewusst designte Funktion.

Memory-Routing am Beispiel

// Nutzer: "Stör mich nie wieder mit Wetter-Updates." → Schicht 01: neue Regel "Themen-Filter: Wetter → unterdrücken" → Schicht 04: optional, als Kontext warum // Nutzer: "Marc hat einen neuen Job bei Bosch." → Schicht 02: Entitätskarte Marc, Feld "Arbeitgeber" = Bosch → Schicht 04: Aussage mit Zeitstempel // Mail kommt rein zu laufendem Projekt → Schicht 03: Projekt-Status "neuer Input erhalten" → Schicht 02: Personen-Karte Absender, "letzte Interaktion" aktualisiert → Schicht 04: Mail-Kontext abgelegt
— 05

Self-Extension

Der Agent kann sich selbst erweitern. Zwei Phasen mit klarem Übergang dazwischen. Phase 1 ist Standardbetrieb, Phase 2 wird bei Bedarf temporär freigeschaltet.

Phase 1 — Tool-Erweiterung (Standardbetrieb)

Der Agent baut sich neue Funktionen die innerhalb des bestehenden Habitats operieren. Greift auf APIs zu, verarbeitet Daten, erweitert seine Fähigkeiten. Keine neuen Pakete, keine Hintergrund-Dienste, keine System-Konfiguration.

Phase 2 — Habitat-Erweiterung (Auftragsbetrieb)

Wenn der Nutzer eine größere Aufgabe gibt die mit Tools nicht lösbar ist, fordert der Agent erweiterte Befugnisse auftragsgebunden an. Darf dann tiefer ins System eingreifen – Pakete installieren, Dienste anlegen. Nach Auftragsende kehrt er in Standardbetrieb zurück. Befugnis ist nicht permanent.

// Phase-2-Beispiel Nutzer: "Sortiere alle Familienfotos der letzten 10 Jahre nach Ereignissen mit Gesichtserkennung." Agent: "Für diese Aufgabe brauche ich: - Gesichtserkennungs-Modell (lokal) - Datenbank für Bild-Embeddings - Hintergrund-Worker für Batch-Verarbeitung Geschätzte Verarbeitungsdauer: ~6 Stunden Geschätzter Speicherbedarf: ~12 GB Genehmigung erteilen?"

Bedarfs-Erkennung: Vorschlags-Modus

Im Reflection-Loop führt der Agent eine Schwächen-Liste – Situationen in denen er etwas nicht oder schlecht konnte. Wenn ein Eintrag eine bestimmte Häufigkeit erreicht, generiert er einen Vorschlag und legt ihn in der UI vor. Nicht den Code – die Idee.

"Mir ist aufgefallen, dass du regelmäßig nach Spritpreisen fragst. Soll ich mir ein Tool dafür bauen das die Tankerkönig-API nutzt? Es würde nichts schreiben, nur lesen. Geschätzte Entwicklungszeit: 30 Sekunden."

Lebenszyklus eines Tools

vorgeschlagen genehmigt · in Entwicklung bereit · wartet auf Aktivierung aktiv verfallen · lange ungenutzt deaktiviert gelöscht

Versionierung: alle Versionen behalten

Wenn der Agent ein Tool verbessert, wird die alte Version nicht überschrieben sondern archiviert. Genau eine Version ist zu jedem Zeitpunkt als aktiv markiert. Vorteile: Rollback möglich, Versions-Vergleich für Lernen, Dokumentations-Spur über Zeit.

Phase-2-Spezifika

Auftragsgebundene Genehmigung – nicht "darf jetzt Pakete installieren" sondern "für diesen Auftrag darf er". Nach Auftrag fällt die Befugnis weg.

Veränderungs-Buchhaltung – jede Veränderung am System wird transaktional protokolliert. "Was hat der Agent in der letzten Woche geändert?" muss beantwortbar sein. Rollback-fähig.

Definiertes Ende – nach Auftragsabschluss fragt der Agent: "Diese Komponenten habe ich gebaut. Behalten, archivieren oder zurückbauen?"

— 06

Architektonische Trennung

Kernwelt und Erweiterungs-Welt liegen in strikt getrennten Verzeichnissen. Diese Grenze ist die wichtigste architektonische Entscheidung des Projekts.

Kernwelt

Der Agent als Produkt. Code von Menschen geschrieben. Stabil, geprüft, getestet. Ändert sich nur durch echte Updates. Der Agent darf hier nicht schreiben.

Erweiterungs-Welt

Separater Ordner mit allem Selbst-Geschaffenen. Tools, Versionen, Specs, Buchhaltung, Logs. Hier darf der Agent vollständig schalten und walten.

Was diese Trennung erkauft

Sauberes Update-Verhalten – Kern wird ersetzt, Erweiterungen bleiben unberührt.

Klare Backup-Strategie – Erweiterungs-Ordner ist die "Persönlichkeit", separat sicherbar. Beim Umzug auf andere Hardware: Ordner mitnehmen, Kern frisch installieren. Genau die Portabilität von Prinzip 7.

Eingrenzung von Schaden – Falls der Agent Tools baut die Probleme machen, ist klar wo sie leben. Ganzer Erweiterungs-Ordner kann in Quarantäne oder gelöscht werden, ohne Kern zu beschädigen.

Lesbarkeit für den Nutzer – Was hat der Agent dazugelernt? Antwort: alles in genau diesem einen Ordner.

Klarheit für den Agent selbst – Er weiß wo seine Grenzen sind. Ordnerzugehörigkeit klärt was "seins" ist.

Was in den Erweiterungs-Ordner gehört

Nicht nur selbstgebauter Code, sondern alle Artefakte der Selbst-Erweiterung:

extensions/ ├── tools/ │ ├── fuel_price_checker/ │ │ ├── v0.1/ (archiviert) │ │ ├── v0.2/ (aktiv) │ │ └── spec.md │ └── time_tracker/ ├── specifications/ (Tool-Specs in lesbarer Form) ├── changelog/ (was wurde wann installiert/geändert) ├── weakness_log/ (Schwächen-Liste für Vorschlags-Modus) ├── usage_logs/ (Tool-Aufrufe, Häufigkeit, Fehler) └── phase2_jobs/ (Auftragsbezogene Komponenten)

Damit ist dieser Ordner faktisch eine fünfte Schicht des Gedächtnisses – eine Schicht aus Code statt aus Daten. Wenn das Gedächtnis gesichert wird, muss dieser Ordner mit. Wenn das Gedächtnis komplett gelöscht wird, sollte der Ordner auch geleert werden – sonst hat der Agent Tools ohne Kontext.

Die Erweiterungs-Welt ist das Tagebuch der Eigenständigkeit des Agenten.
— 07

Implementierungsplan

Sechs Etappen, vertikal gedacht: nach jeder steht ein lauffähiges System. Der Agent wird nicht in Schichten gebaut, sondern wächst als Wesen – erst Stimme, dann Innenwelt, dann Zeitgefühl, dann Hände, dann Eigenleben, dann ein Gesicht.

Leitprinzip: vertikal statt horizontal

Statt erst alle Grundlagen und dann alle Features zu bauen, bekommt der Agent in jeder Etappe eine qualitativ neue Eigenschaft. Das hat drei konkrete Vorteile: nach jeder Etappe ist etwas tatsächlich nutzbar. Konzeptionelle Annahmen werden früh getestet und können korrigiert werden, bevor zu viel darauf aufbaut. Und das Schwerste (Reflection-Loop, Bewertungs-Pipeline) kommt erst spät, wenn das Fundament steht.

ETAPPE 01 Stimme & Ohren

Du redest mit dem Agent, er redet zurück. Sonst nichts.

Das absolute Fundament. Sprache wird aufgenommen, transkribiert, an Ollama geschickt, die Antwort wird gesprochen. Was hier entsteht: STT (Whisper.NET), TTS (Piper), die komplette Audio-Pipeline. Wake Word kommt noch nicht – Aktivierung per Tastendruck oder Schlüsselwort.

Was bewusst noch nicht drin ist: kein Gedächtnis, keine Tools, keine Zeitwahrnehmung. Der Agent vergisst alles sobald die Konversation endet. Er ist ein freundlicher dummer Sprecher.

Konzeptionell entscheidend. Hier wird der Tonfall des Systems geprägt – wie schnell antwortet er, wie klingt seine Stimme. Die Latenz-Disziplin entsteht hier oder gar nicht. Würde Sprache erst spät kommen, wäre das ganze System zu langsam und müsste umgebaut werden.
ETAPPE 02 Gehirn

Der Agent erinnert sich. Er weiß wer du bist.

Jetzt entstehen die strukturierten Gedächtnis-Schichten 1, 2 und 3. SQLite-Schema für Regeln, Entitäten, aktive Themen. Das Memory-Routing – die Logik die beim Eintreffen entscheidet was wo landet. Und die Lese-Operationen: der Agent durchsucht sein Gedächtnis und nutzt es im Gespräch.

Erst ab hier ist der Agent persönlich. Du erzählst ihm etwas, beim nächsten Gespräch erinnert er sich. Personen-Karten füllen sich. Aktive Themen rotieren rein und raus.

Was bewusst noch nicht drin ist: Vector-DB / Schicht 4. Bis dahin lebt das episodische Gedächtnis als simples Gesprächs-Log mit Volltextsuche – das reicht für diese Etappe.

Konzeptionell entscheidend. Das Memory-Routing erzwingt viele Detail-Entscheidungen die wir nur abstrakt diskutiert haben. Was ist eine Regel vs. eine beiläufige Bemerkung? Wann wird eine Personen-Karte angelegt? Wann konsolidiert sich Information? Diese Fragen werden hier konkret beantwortet.
ETAPPE 03 Zeitgefühl

Der Agent versteht Zukunft. Er kann Termine planen und einhalten.

Der Scheduler entsteht (Quartz.NET mit persistentem Job-Store), und das erste Tool-Aufruf-System überhaupt. Der Agent bekommt sein erstes Werkzeug: ein Tool das einen Job in seine Zukunft einplanen kann.

Du sagst "weck mich morgen um 7" – passiert. "Erinner mich in 20 Minuten" – passiert. Der Agent weiß plötzlich dass es einen Moment gibt in dem er etwas tun wird. Und wegen Etappe 2 weiß er warum – er kann den Auftrag in deinen Kontext einbetten.

Hier entsteht außerdem das Pattern für alle weiteren Tools: wie definiert das Backend eine Funktion, wie reicht es das Schema an das LLM, wie ruft es nach Antwort die Funktion auf, wie speist es das Ergebnis zurück. Diese Schiene fährt später alles andere.

Konzeptionell entscheidend. Der Scheduler ist die Voraussetzung für alles Proaktive das später kommt. Reflection-Loop, Erinnerungen, verzögerte Aktionen – alles sind im Kern geplante Tasks. Wenn der Scheduler hier solide steht, ist die Mechanik für vieles spätere bereits da.
ETAPPE 04 Fähigkeiten

Der Agent greift in die Welt. Tool für Tool, eines nach dem anderen.

Jetzt kommen die Tools. Aber nicht alle gleichzeitig – nacheinander, jedes vollständig durchgedrückt bevor das nächste angefangen wird. Reihenfolge:

Erst die Lese-Tools: Kalender (CalDAV), dann Mail (IMAP, ein Postfach), dann ein Messenger nach Wahl. Dann das erste Schreib-Tool mit dem Vorlage-Mechanismus (Mail-Entwurf vorbereiten zur Freigabe). Dann Home Assistant mit Auto-Discovery. Dann Web-Suche über SearXNG.

Am Ende dieser Etappe weißt du sehr gut welche Tools dir noch fehlen. Das ist Material für die spätere Self-Extension.

Konzeptionell entscheidend. In dieser Etappe wird der Vorlage-Mechanismus geboren – der Knoten den wir vertagt hatten. Sobald das erste Schreib-Tool entsteht, brauchst du eine Antwort darauf wo Vorlagen leben und wie du sie freigibst. Bis zur richtigen UI ist das pragmatisch eine Datei in einem speziellen Verzeichnis – hässlich, aber funktional.
ETAPPE 05 Eigenleben

Der Agent denkt von selbst nach. Er meldet sich – oder schweigt.

Jetzt erst kommt der Reflection-Loop und die vierstufige Bewertungs-Pipeline. Der Agent beobachtet die Quellen die in Etappe 4 angeschlossen wurden. Er bewertet ob etwas wert ist dich zu unterbrechen. Stille ist der Standardzustand.

Diese Etappe ist die längste und experimentellste. Du wirst Schwellenwerte feinjustieren, Regeln eintragen, dem Agent über Tage und Wochen beibringen was wichtig ist. Das war geplant – die Lernschleife funktioniert nur durch Nutzung.

Output erstmal direkt über Lautsprecher (wenn zuhause) oder als System-Benachrichtigung. Die richtige UI kommt erst danach.

Konzeptionell entscheidend. Hier wird die hybride Pipeline (deterministische Filter + LLM-Bewertung) auf die Probe gestellt. Wenn die Annahmen aus den Konzeptgesprächen stimmen, funktioniert sie. Wenn nicht, wird hier nachgebessert.
ETAPPE 06 Sichtbarkeit

Die React-UI. Mit voller Kenntnis was sie zeigen muss.

Jetzt kommt das Frontend. Status, Chat, Anrufen, Extensions, Gedächtnis, Gedanken, Quellen – die Tabs aus dem Mockup. Aber gebaut nicht für vermutete sondern für erfahrene Bedürfnisse, weil das Backend seit Wochen läuft und du es benutzt hast.

Die UI wird wahrscheinlich kleiner und anders als ursprünglich gedacht – manche Dinge werden sich als unwichtig erwiesen haben, andere als wichtiger. Das ist gut so. Eine UI die aus Erfahrung wächst statt aus Spekulation ist immer besser.

Konzeptionell entscheidend. Bis zu dieser Etappe nutzt du den Agent über Stimme und rohe Hilfsmittel (Logs, Datei-Vorlagen, System-Notifications). Diese Übergangsphase ist die beste UX-Forschung die du machen kannst – du merkst genau was dir an Information fehlt.

Was nach Etappe 6 kommt

Self-Extension Phase 1 – der Agent baut sich selbst Tools. Erst sinnvoll wenn er eine Weile gelaufen ist und seine Schwächen-Liste gefüllt hat.

Vector-DB / Schicht 4 – wenn genug episodisches Material gesammelt ist um Vector-Suche zu rechtfertigen.

Notifications nach außen – Push, SMS, Telegram. Wenn du den Agent auch unterwegs nutzen willst.

Self-Extension Phase 2 – Habitat-Erweiterung. Der Agent darf das System verändern. Nur wenn alles davor stabil ist.

Der Agent wird nicht in Schichten gebaut. Er wächst als Wesen.

Tech-Stack (Status: vorgesehen)

Backend – .NET 9 mit Microsoft Semantic Kernel als Agent-Framework, Ollama als LLM-Server, Gemma 4 E4B als Modell. Quartz.NET für den Scheduler. SQLite für strukturierte Schichten. Qdrant oder SQLite-vec für Vector-Memory. Roslyn für dynamische Code-Generierung.

Frontend – React-App, kommuniziert mit Backend via REST + WebSocket. Erst in Etappe 6.

Sprache – Whisper.NET für STT, Piper TTS für Sprachausgabe, openWakeWord für Wake-Word-Erkennung (kommt später).

Hardware – Lokales Testing auf Entwickler-Maschine. Später: AMD Ryzen Mini-PC mit 32 GB DDR5. Portabel als Container.

— 08

Was noch konzeptionell offen ist

Diese Punkte sind bewusst noch nicht entschieden. Sie sollten vor oder während der Implementierung geklärt werden.

Konzept-Lücken

  • Vorlage-Mechanik – wie eine vom Agent vorbereitete Aktion (Mail, Antwort, Termin) beim Nutzer ankommt. Wo lebt die Vorlage, wie wird sie präsentiert, was passiert wenn sie liegen bleibt? Wird konkret in Etappe 4.
  • Vergessen als Funktion – Verfall, Konsolidierung, bewusstes Löschen. Wurde vertagt aber wird relevant sobald das Gedächtnis wächst.
  • Identität & Persönlichkeit – Name, Tonfall, Ansprache. Wurde vertagt bis die Features stehen, aber wird in Etappe 1 spürbar.
  • Vector-DB-Interaktion in der UI – wie der Nutzer Schicht 4 durchsucht und einsehen kann. Wurde der UI-Phase zugewiesen.
  • Eigenes Wissen einspielen – Datei/Ordner-Upload des Nutzers in die Vector-DB. UI-Phase.
  • Memory-Routing-Detail – wie der Agent zwischen "Regel" und "beiläufige Bemerkung" unterscheidet. Wird in Etappe 2 konkret.
  • Notifications nach außen – Push, SMS, Telegram. Erstmal vertagt.
